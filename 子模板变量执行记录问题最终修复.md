# 子模板变量执行记录问题最终修复

## 问题回顾

用户报告了两个关键问题：

1. **`/generate/task_xxx` 页面**: 看得到主模板的变量执行情况，但看不到子模板的变量执行情况
2. **`/logs/task_xxx` 页面**: "所属模板"列全部显示主模板

## 问题根本原因

### 原因 1: 数据库字段缺失
代码中添加了 `template_id` 和 `template_path` 字段，但数据库表中还没有这些列。

**错误信息**:
```
Unknown column 'template_id' in 'field list'
```

### 原因 2: progress_callback 的元数据查找问题
在 `backend/app/api/reports.py` 中，主模板的 `progress_callback` 使用 `parsed_metadata` (主模板的元数据) 来查找变量信息：

```python
async def progress_callback(var_name: str, status: VariableStatus, result):
    var_meta = parsed_metadata[var_name]  # ❌ 这里有问题！
    # ...
```

当子模板的变量（如 `title1`, `content1`）执行时，`progress_callback` 尝试从主模板的 `parsed_metadata` 中查找它们，但主模板的元数据中只有 `title` 和 `content`，所以抛出 `KeyError`，导致：

- ✗ 子模板的变量不会被记录到数据库
- ✗ 不会通过 WebSocket 广播
- ✗ 前端看不到子模板的变量

### 步骤 2: 为子模板创建专用的 progress_callback

修改了 `backend/app/services/renderer.py` 的 `_execute_and_render_template` 方法，为子模板创建专用的 `progress_callback`：

```python
# 为子模板创建专用的 progress_callback
# 直接使用子模板的元数据写入数据库，不依赖主模板的 progress_callback
async def sub_template_progress_callback(var_name: str, status, result):
    if not progress_callback:
        return
    
    # ✓ 从子模板的 metadata 获取变量元数据
    if var_name not in metadata:
        logger.warning(f"变量 {var_name} 不在子模板 {template_id} 的元数据中")
        return
    
    var_meta = metadata[var_name]  # ✓ 使用子模板的元数据！
    
    # 导入必要的模块
    from app.models.db_models import GenerationTaskVariable, VariableStatusType
    from app.services.websocket_manager import ws_manager
    from app.database import SessionLocal
    from datetime import datetime
    from app.core.models import VariableStatus
    
    db_session = SessionLocal()
    try:
        if status == VariableStatus.RUNNING:
            var_record = GenerationTaskVariable(
                task_id=main_task_id,
                variable_name=var_name,
                source=var_meta.source.value,
                status='running',
                started_at=datetime.utcnow(),
                template_id=template_id,        # ✓ 子模板的ID
                template_path=template_path     # ✓ 完整的层级路径
            )
            db_session.add(var_record)
            db_session.commit()
            
            # ✓ 广播 WebSocket 事件
            await ws_manager.broadcast_variable_started(...)
        elif status in (VariableStatus.SUCCESS, VariableStatus.FAILED):
            # ✓ 更新记录
            var_record = db_session.query(GenerationTaskVariable).filter(...).first()
            # ...
            db_session.commit()
            
            # ✓ 广播完成/失败事件
            await ws_manager.broadcast_variable_completed/failed(...)
    except Exception as e:
        logger.error(f"子模板变量进度回调失败: {e}", exc_info=True)
    finally:
        db_session.close()

# ✓ 传递子模板专用的 progress_callback
await scheduler.execute_all(context, sub_template_progress_callback if progress_callback else None)
```

## 技术要点

### 1. 子模板 progress_callback 的设计
- **独立性**: 子模板有自己的 progress_callback，不依赖主模板
- **元数据来源**: 从子模板的 `metadata` 获取变量信息
- **模板信息**: 使用子模板的 `template_id` 和 `template_path`
- **数据库会话**: 创建独立的数据库会话，避免冲突

### 2. 错误处理
- 使用 try-except 捕获异常，避免子模板变量执行失败导致整个流程中断
- 记录详细的错误日志，便于调试

### 3. WebSocket 广播
- 子模板的变量执行也会通过 WebSocket 实时广播
- 前端可以实时看到子模板的变量执行进度
