# 子模板变量执行记录修复总结

## 问题描述

用户反馈了两个关键问题：

1. **执行日志中所有的所属模板都显示为主模板** - 子模板的日志没有正确标记
2. **在 `/generate/task_xxx` 界面上的变量执行情况没有显示子模板的变量执行情况** - 前端看不到子模板的变量

## 根本原因

### 数据库迁移缺失
在之前的更新中，我们在代码中添加了 `template_id` 和 `template_path` 字段，但数据库表中还没有这些列，导致：
- 错误信息：`Unknown column 'template_id' in 'field list'`
- 报告生成失败，无法显示任何变量执行情况

### 子模板变量执行未记录
在 `renderer.py` 的 `_execute_and_render_template` 方法中，子模板执行变量时调用：
```python
await scheduler.execute_all(context)  # 没有传递 progress_callback
```

而主模板执行时调用：
```python
await scheduler.execute_all(context, progress_callback)  # 有 progress_callback
```

这导致：
1. ✗ 子模板的变量执行**没有记录到数据库**（`generation_task_variables` 表）
2. ✗ 子模板的变量执行**没有通过 WebSocket 广播**
3. ✗ 前端界面**看不到子模板的变量执行情况**

### 2. 传递 progress_callback 到子模板（已完成）

#### 修改文件：`backend/app/services/renderer.py`

**步骤 1：** 在 `_resolve_includes` 方法签名中添加 `progress_callback` 参数

```python

    self,
    template_content: str,
    db_session: Session,
    nested_user_inputs: Dict[str, Dict[str, Any]],

```

**步骤 2：** 在递归调用时传递 `progress_callback`

```python
resolved_sub_content = await self._resolve_includes(
    sub_template.template_content,

    nested_user_inputs,
```

## 技术要点

### 1. Progress Callback 的传递链
```
reports.py (主模板执行)
  └─> progress_callback 定义
       └─> _resolve_includes 
            └─> _execute_and_render_template (子模板执行)
                 └─> scheduler.execute_all(context, progress_callback)
                      └─> 记录变量执行到数据库
                      └─> 广播 WebSocket 事件
```

### 2. 模板路径构建
- 主模板：`template_path = "主模板名称"`
- 一级子模板：`template_path = "主模板 > 子模板1"`
- 二级子模板：`template_path = "主模板 > 子模板1 > 子模板2"`

### 3. Context 的 template_id 和 template_path
每个模板（主模板和子模板）都有自己的 `ExecutionContext`，其中：
- `context.task_id` - 所有模板共享同一个主任务ID
- `context.template_id` - 当前模板的ID
- `context.template_path` - 当前模板的完整层级路径

这些信息会在 `progress_callback` 中被读取并记录到数据库。
