# 枚举问题紧急修复方案

## 问题总结

**核心问题**：SQLAlchemy 的 ENUM 类型处理与 MySQL ENUM 存在兼容性问题

**错误信息**：
```
LookupError: 'api' is not among the defined enum values. 
Enum name: variablesourcetype. 
Possible values: USER_INPUT, SQL, API, ..., VISION_AI
```

**影响范围**：
- 无法查看任何任务的生成过程
- 报告列表可能显示异常
- 部分 API 接口返回 500 错误

## 已尝试的修复

1. ✅ 清理异常数据（338条）
2. ✅ 重启后端服务（多次）
3. ✅ 同步枚举定义（core/models, db_models, frontend）
4. ✅ 数据库 ENUM 迁移（添加 'constant'）
5. ✅ 将 ORM 枚举列改为字符串类型
6. ✅ 修改 API 使用原生 SQL 查询

**但问题仍然存在！**

---

## 根本原因分析

1. **数据库层面**：MySQL ENUM 定义正确（'api', 'user_input', 'constant'）
2. **Python 层面**：枚举定义正确（API = "api", USER_INPUT = "user_input"）
3. **SQLAlchemy 层面**：在读取时将数据库值转换为 Python 枚举对象时失败

**问题出在**：
- SQLAlchemy 缓存了旧的表元数据
- 即使修改了 model 定义，metadata 缓存仍在
- `--reload` 模式无法清除 SQLAlchemy 的内部缓存

### 方案 B：创建单独的只读 API 绕过 ORM

创建一个新的 API 端点，完全不使用 ORM：

```python
# 在 reports.py 中添加
@router.get("/tasks/{task_id}/status-raw")
async def get_task_status_raw(task_id: str):
    """使用原生SQL获取任务状态（绕过ORM）"""
    from sqlalchemy import create_engine, text
    from app.database import DATABASE_URL
    
    engine = create_engine(DATABASE_URL)
    with engine.connect() as conn:
        # 查询任务
        task = conn.execute(text("""
            SELECT id, template_id, status, inputs_json,
                   started_at, finished_at, created_at
            FROM generation_tasks WHERE id = :id
        """), {"id": task_id}).fetchone()
        
        if not task:
            return {"error": "Task not found"}, 404
        
        # 查询变量
        variables = conn.execute(text("""
            SELECT variable_name, source, status, started_at, 
                   finished_at, duration_ms, error_message, result_preview
            FROM generation_task_variables 
            WHERE task_id = :id
            ORDER BY id
        """), {"id": task_id}).fetchall()
        
        # 查询报告
        report = conn.execute(text("""
            SELECT id FROM reports WHERE task_id = :id
        """), {"id": task_id}).fetchone()
        
        return {
            "task_id": task[0],
            "template_id": task[1],
            "status": task[2],
            "inputs_json": task[3],
            "started_at": str(task[4]) if task[4] else None,
            "finished_at": str(task[5]) if task[5] else None,
            "created_at": str(task[6]) if task[6] else None,
            "report_id": report[0] if report else None,
            "variables": [
                {
                    "variable_name": v[0],
                    "source": v[1],
                    "status": v[2],
                    "started_at": str(v[3]) if v[3] else None,
                    "finished_at": str(v[4]) if v[4] else None,
                    "duration_ms": v[5],
                    "error_message": v[6],
                    "result_preview": v[7]
                }
                for v in variables
            ]
        }
```

### 方案 C：数据库级别修复（彻底解决）

**原理**：将数据库的 ENUM 列改为 VARCHAR

```sql
-- 备份数据库
mysqldump -u root -p123456 -h 10.10.20.10 -P 24406 md_agent > backup_enum_fix.sql

-- 修改列类型
ALTER TABLE generation_task_variables 
MODIFY COLUMN source VARCHAR(50) NOT NULL,
MODIFY COLUMN status VARCHAR(50) NOT NULL;

ALTER TABLE generation_tasks
MODIFY COLUMN status VARCHAR(50) NOT NULL;

ALTER TABLE reports
MODIFY COLUMN status VARCHAR(50) NOT NULL;

-- 验证
SHOW COLUMNS FROM generation_task_variables LIKE 'source';
SHOW COLUMNS FROM generation_task_variables LIKE 'status';
```

**优点**：
- 彻底解决枚举问题
- 不需要修改 Python 代码（已经改为 String）
- 数据库和代码完全一致

**缺点**：
- 需要数据库权限
- 修改表结构（但数据不受影响）
- 需要在生产环境谨慎执行
