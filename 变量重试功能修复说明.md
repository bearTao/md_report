# 变量重试功能修复说明

**修复日期**: 2025-10-23  
**版本**: v1.0

---

## 问题背景

### 发现的问题

1. **后端问题**：变量重试后状态变为PENDING，但重试任务没有被执行
2. **前端问题**：重复点击重试按钮导致400错误
3. **用户体验问题**：错误提示不够友好

### 具体现象

- 用户点击重试按钮后，变量状态变为"等待中"
- 刷新页面后发现变量一直处于等待状态，没有实际执行
- 多次点击重试按钮返回400错误：`Cannot retry variable in status: pending`

### 根本原因

1. **后端**：`retry_variable_execution`是async函数，但通过FastAPI的`BackgroundTasks.add_task`调用时无法正确执行异步代码
2. **前端**：缺少重复点击防护和状态检查

---

## 修复方案

### 1. 后端修复

**文件**: `/data/tao/code/xuqiu/backend/app/api/reports.py`

**修改内容**:

将后台任务的启动方式从`BackgroundTasks.add_task`改为`asyncio.create_task`，确保异步函数能够正确执行。

**修改前**:
```python
# Trigger retry in background
background_tasks.add_task(
    retry_variable_execution,
    task_id=task_id,
    variable_name=variable_name
)
```

**修改后**:
```python
# Check if variable can be retried
# Allow retry for FAILED, CANCELLED, and stuck PENDING (no started_at or started > 5 min ago)
can_retry = False

if var.status in [VariableStatusType.FAILED, VariableStatusType.CANCELLED]:
    can_retry = True
elif var.status == VariableStatusType.PENDING:
    # Allow retry if stuck in PENDING (task might be lost due to server restart)
    if var.started_at is None:
        can_retry = True
    else:
        # If started more than 5 minutes ago but still pending, consider it stuck
        time_since_start = (datetime.utcnow() - var.started_at).total_seconds()
        if time_since_start > 300:  # 5 minutes
            can_retry = True

if not can_retry:
    raise HTTPException(
        status_code=400,
        detail=f"Cannot retry variable in status: {var.status.value}. Variable is currently running or completed."
    )

# Reset variable status and trigger retry
var.status = VariableStatusType.PENDING
var.started_at = None
# ... reset other fields ...
db.commit()

# Trigger retry in background using asyncio.create_task
# This ensures the async function is properly executed
asyncio.create_task(retry_variable_execution(task_id, variable_name))
```

**技术说明**:
- FastAPI的`BackgroundTasks.add_task`只支持同步函数或通过特殊包装的异步函数
- 直接使用`asyncio.create_task`可以在当前事件循环中创建异步任务
- 移除了WebSocket广播（API响应和后续的变量执行开始事件已足够通知前端）

**额外修复 - WebSocket广播参数**:
在`retry_variable_execution`函数中，修复了`broadcast_variable_started`调用缺少参数的问题：
```python
await ws_manager.broadcast_variable_started(
    task_id=task_id,
    variable_name=variable_name,
    source=variable_meta.source.value,
    dependencies=variable_meta.dependencies or [],  # 新增
    started_at=var_record.started_at  # 新增
)
```

**重要改进 - 处理卡住的PENDING状态**:
- 允许重试卡在`PENDING`状态的变量（可能因服务重启导致任务丢失）
- 检查逻辑：
  - 如果`started_at`为空，说明任务从未开始，允许重试
  - 如果`started_at`超过5分钟，说明任务可能卡住或丢失，允许重试
  - 否则认为任务正在执行，拒绝重试

### 2. 前端修复

**文件**: `/data/tao/code/xuqiu/frontend/src/pages/generate/ReportProgress.tsx`

#### 2.1 添加状态检查函数

```typescript
// 检查变量是否可以重试
const canRetry = (status: string) => {
  return status === 'failed' || status === 'cancelled';
};
```

#### 2.2 改进错误处理

**修改前**:
```typescript
onError: (error: any) => {
  message.error(`重试失败: ${error.response?.data?.detail || error.message}`);
}
```

**修改后**:
```typescript
onError: (error: any) => {
  const errMsg = error.response?.data?.detail || error.message;
  // 友好的错误提示
  if (errMsg.includes('status')) {
    message.error('该变量正在执行中或已完成，无法重试');
  } else if (errMsg.includes('not found')) {
    message.error('变量不存在，请刷新页面重试');
  } else {
    message.error(`重试失败: ${errMsg}`);
  }
}
```

#### 2.3 优化重试处理函数

**修改前**:
```typescript
const handleRetryVariable = (variableName: string) => {
  retryVariableMutation.mutate(variableName);
};
```

**修改后**:
```typescript
const handleRetryVariable = (variableName: string, status: string) => {
  if (!canRetry(status)) {
    message.warning('只有失败或已取消的变量才能重试');
    return;
  }
  retryVariableMutation.mutate(variableName);
};
```

#### 2.4 改进按钮禁用逻辑

**修改前**:
```typescript
{variable.status === 'failed' && (
  <Button
    type="link"
    size="small"
    icon={<ReloadOutlined />}
    onClick={() => handleRetryVariable(variable.variable_name)}
    loading={retryVariableMutation.isPending}
  >
    重试
  </Button>
)}
```

**修改后**:
```typescript
{(variable.status === 'failed' || variable.status === 'cancelled') && (
  <Button
    type="link"
    size="small"
    icon={<ReloadOutlined />}
    onClick={() => handleRetryVariable(variable.variable_name, variable.status)}
    loading={retryVariableMutation.isPending}
    disabled={!canRetry(variable.status) || retryVariableMutation.isPending}
  >
    重试
  </Button>
)}
```

#### 2.5 添加cancelled状态支持

在`getStatusTag`函数中添加cancelled状态的显示：

```typescript
const statusMap: Record<string, { color: string; text: string; icon?: React.ReactNode }> = {
  pending: { color: 'default', text: '等待中' },
  running: { color: 'processing', text: '执行中' },
  success: { color: 'success', text: '成功' },
  failed: { color: 'error', text: '失败' },
  skipped: { color: 'warning', text: '跳过' },
  cancelled: { color: 'default', text: '已取消' },  // 新增
};
```

## 功能改进

### 防止重复点击

1. **按钮禁用**：
   - 重试请求进行中时（`retryVariableMutation.isPending`），按钮禁用
   - 变量状态不是`failed`或`cancelled`时，按钮禁用

2. **前置检查**：
   - 在`handleRetryVariable`函数中添加状态检查
   - 不符合条件时直接返回并显示警告消息

### 友好的错误提示

根据错误类型显示不同的中文提示：

| 错误类型 | 提示消息 |
|---------|---------|
| 状态不允许 | "该变量正在执行中或已完成，无法重试" |
| 变量不存在 | "变量不存在，请刷新页面重试" |
| 其他错误 | "重试失败: {具体错误信息}" |

### 状态显示优化

- 为所有可能的变量状态提供清晰的中文标签
- 新增对`cancelled`状态的支持
- `pending`状态显示"等待中"，帮助用户理解当前状态

## 注意事项

### 后端

1. **异步任务执行**：
   - `asyncio.create_task`在当前事件循环中创建任务
   - 确保事件循环正在运行（FastAPI自动管理）
   - 任务失败不会影响API响应
   - ⚠️ **重要**：如果服务重启，未完成的异步任务会丢失

2. **数据库会话**：
   - 后台任务使用独立的数据库会话（`SessionLocal()`）
   - 任务结束后正确关闭会话

3. **卡住的PENDING状态处理**：
   - 服务重启可能导致任务丢失，变量卡在PENDING状态
   - 后端会自动检测并允许重试卡住的变量
   - 判断标准：`started_at`为空或超过5分钟
   
4. **手动修复卡住的变量**（如需要）：
   ```python
   cd /data/tao/code/xuqiu/backend
   python -c "
   from app.database import SessionLocal
   from app.models.db_models import GenerationTaskVariable, VariableStatusType
   
   db = SessionLocal()
   var = db.query(GenerationTaskVariable).filter(
       GenerationTaskVariable.task_id == 'your_task_id',
       GenerationTaskVariable.variable_name == 'your_variable_name'
   ).first()
   
   var.status = VariableStatusType.FAILED
   var.error_message = 'Task lost due to server restart'
   db.commit()
   db.close()
   "
   ```

### 前端

1. **状态同步**：
   - 使用React Query的`refetch`确保状态及时刷新
   - WebSocket连接可以实时接收状态更新

2. **按钮状态管理**：
   - `disabled`属性确保按钮不可点击
   - `loading`属性提供视觉反馈

3. **错误处理**：
   - 所有API错误都应该有友好的中文提示
   - 区分不同类型的错误，提供针对性的解决建议
