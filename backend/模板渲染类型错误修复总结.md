# 模板渲染类型错误修复总结

## 修复日期
2025-10-20

## 问题描述

用户报告：所有变量执行成功，但模板渲染时失败。前端只显示"报告生成失败，请查看下方变量执行详情"，但**没有显示具体的错误原因**，而实际上错误与变量执行无关，是**模板渲染问题**。

### 后端错误日志
```
TypeError: must be real number, not str
  File "<template>", line 92, in top-level template code
  File ".../jinja2/filters.py", line 1016, in do_format
    return soft_str(value) % (kwargs or args)
```

### 根本原因

**数据库返回的数字字段可能是字符串类型**，而Jinja2模板中直接使用了需要数字类型的操作（format、sum、sort等），导致类型不匹配错误。

**具体问题**：
1. 第92行：`"%.2f"|format(value)` - value是字符串
2. 第97行：`index_scores | sum(attribute='index_deduction')` - 字符串和数字混合
3. 第100行：`index_scores | sort(attribute='index_deduction')` - 无法比较字符串和数字

## 修复内容

### 1. 修复模板中的类型问题

**文件**: 数据库中的模板（`tpl_ai_mysql`, `tpl_f202f946b0da`）

#### a) 添加数据预处理代码

在模板开始处添加数据清洗代码，确保所有数字字段都是float类型：

```jinja2
{# 数据预处理：确保所有数字字段都是数字类型 #}
{% set index_scores_cleaned = [] %}
{% for score in index_scores %}
  {% set _ = index_scores_cleaned.append({
    'index': score.get('index', ''),
    'values': (score.get('values') or score.get('value_data') or 0) | float,
    'value_data': (score.get('value_data') or score.get('values') or 0) | float,
    'index_weight': (score.get('index_weight') or 0) | float,
    'index_score': (score.get('index_score') or 0) | float,
    'index_deduction': (score.get('index_deduction') or 0) | float
  }) %}
{% endfor %}
{% set index_scores = index_scores_cleaned %}
```

**作用**：
- 遍历所有`index_scores`项
- 将每个数字字段转换为float类型
- 创建新的"cleaned"列表
- 替换原始列表

#### b) 修复第92行的format问题

**修复前**:
```jinja2
{{ "%.2f"|format(idx_value) }}
```

**修复后**:
```jinja2
{{ "%.2f"|format(idx_value|float) if idx_value is not none and idx_value != '' else '-' }}
```

**说明**: 添加`|float`过滤器，在格式化前先转换类型。

#### c) 修复第97行的sum问题

**修复前**:
```jinja2
{% set total_deduction = index_scores | sum(attribute='index_deduction', start=0) %}
```

**修复后**:
```jinja2
{% set total_deduction = index_scores | map(attribute='index_deduction') | map('float') | sum %}
```

**说明**: 使用`map('float')`先转换所有值为float，再求和。

**注意**: 由于添加了数据预处理，这一步实际上已经不需要了，但保留以确保兼容性。

## 前端错误显示问题

### 问题分析
用户反馈前端没有显示详细的错误信息，只显示：
```
报告生成失败
请查看下方变量执行详情了解失败原因
```

但实际上这个错误**与变量执行无关**，是**模板渲染错误**。

### 原因
后端已经通过WebSocket发送了详细的错误信息（`error_info`），但前端可能：
1. 没有正确监听`task_failed`事件
2. 没有显示`error`字段中的详细信息
3. 错误消息被通用提示覆盖

### 后端已做的工作
1. ✅ 通过`_format_error_details`格式化详细错误
2. ✅ 通过`ws_manager.broadcast_task_failed`发送错误
3. ✅ 错误信息包含：
   - `code`: 错误代码（如`TEMPLATE_RENDER_ERROR`）
   - `message`: 友好的错误消息
   - `details`: 详细信息（类型、行号、字段等）
   - `suggestion`: 修复建议
   - `traceback`: 堆栈跟踪（最后10行）

### WebSocket事件结构
```javascript
{
  "type": "task_failed",
  "data": {
    "error": {
      "code": "TEMPLATE_RENDER_ERROR",
      "message": "模板渲染失败：格式化错误（得到str类型，期望数字）",
      "details": {
        "type": "format_type_error",
        "description": "格式化时类型不匹配，期望数字但得到字符串",
        "wrong_type": "str",
        "error": "must be real number, not str",
        "traceback": [...]
      },
      "suggestion": "在格式化前先转换类型：{{ '%.2f'|format(value|float) }}"
    },
    "summary": {
      "duration_ms": 16000
    }
  }
}
```

### 前端需要做的改进

#### 1. 监听task_failed事件
```javascript
ws.on('message', (data) => {
  if (data.type === 'task_failed') {
    // 显示详细错误信息
    showDetailedError(data.data.error);
  }
});
```

#### 2. 显示详细错误
```javascript
function showDetailedError(error) {
  // 显示错误代码
  console.log('Error Code:', error.code);
  
  // 显示友好的错误消息
  alert(error.message);
  
  // 在UI中显示详细信息
  if (error.details) {
    showErrorDetails({
      type: error.details.type,
      description: error.details.description,
      line: error.details.line,  // 模板行号
      field: error.details.field  // 问题字段
    });
  }
  
  // 显示修复建议
  if (error.suggestion) {
    showSuggestion(error.suggestion);
  }
}
```

#### 3. 区分错误类型
```javascript
// 根据错误代码显示不同的UI
switch (error.code) {
  case 'TEMPLATE_RENDER_ERROR':
    // 模板渲染错误 - 显示模板相关的提示
    showTemplateError(error);
    break;
  
  case 'SQL_EXECUTION_ERROR':
    // SQL错误 - 显示数据库相关的提示
    showSQLError(error);
    break;
  
  case 'AI_GENERATION_ERROR':
    // AI错误 - 显示AI相关的提示
    showAIError(error);
    break;
  
  default:
    // 通用错误
    showGenericError(error);
}
```

## 根本解决方案（可选）

### 方案：在DatabaseConnector中统一处理类型

当前修复是在模板层面处理类型转换，但更好的方案是在**数据层**确保类型正确。

#### 修改 `app/connectors/database.py`

```python
def _convert_to_serializable(self, obj: Any) -> Any:
    """Convert non-JSON-serializable objects to serializable format"""
    if isinstance(obj, (datetime, date)):
        return obj.isoformat()
    elif isinstance(obj, Decimal):
        return float(obj)  # 确保Decimal转为float，而不是str
    elif isinstance(obj, (int, float)):
        return obj  # 数字类型保持原样
    elif isinstance(obj, str):
        # 尝试转换数字字符串
        try:
            # 检查是否是数字字符串
            if '.' in obj:
                return float(obj)
            else:
                return int(obj)
        except ValueError:
            return obj  # 不是数字，保持字符串
    # ... 其他类型处理
```

**优点**：
- 一次修复，所有模板受益
- 数据类型更统一
- 减少模板复杂度

**缺点**：
- 可能影响其他依赖字符串类型的代码
- 需要全面测试

**建议**: 当前的模板层修复已经足够，除非发现大量模板都有类似问题，否则不建议修改DatabaseConnector。

## 最佳实践建议

### 1. 编写模板时

#### ✅ 推荐做法
```jinja2
{# 在使用前先转换类型 #}
{{ "%.2f"|format(value|float) }}

{# 使用默认值处理None #}
{{ (value or 0)|float }}

{# 安全的条件判断 #}
{% if value and value|float > 0 %}
  ...
{% endif %}
```

#### ❌ 避免做法
```jinja2
{# 直接格式化，可能是字符串 #}
{{ "%.2f"|format(value) }}

{# 直接sum，可能类型混合 #}
{{ items | sum(attribute='count') }}

{# 直接比较，可能类型不同 #}
{% if value > 10 %}
```

### 2. 处理数字字段
```jinja2
{# 方案1: 在模板开始时预处理 #}
{% set cleaned_data = [] %}
{% for item in data %}
  {% set _ = cleaned_data.append({
    'name': item.name,
    'value': (item.value or 0) | float
  }) %}
{% endfor %}

{# 方案2: 使用时转换 #}
{{ "%.2f"|format((item.value or 0)|float) }}

{# 方案3: 使用default过滤器 #}
{{ item.value | default(0) | float }}
```

### 3. 处理聚合操作
```jinja2
{# sum - 先map转换 #}
{% set total = items | map(attribute='value') | map('float') | sum %}

{# sort - 确保类型一致 #}
{% set sorted_items = items | sort(attribute='value') %}
{# 如果类型可能不一致，先预处理 #}

{# filter - 注意类型 #}
{% set high_values = items | selectattr('value', '>', 10) %}
{# 可能需要先转换: value|float > 10 #}
```
