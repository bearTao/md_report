## 1. 数据库枚举类型修复 ✅

### 问题描述
数据库 `generation_task_variables` 表的 `source` 字段枚举类型不包含 `IMAGE` 和 `VISION_AI`，导致插入新变量类型时报错：
```
DataError: (1265, "Data truncated for column 'source' at row 1")
```

### 修复方案
创建并执行SQL脚本扩展枚举类型：

**文件**: `backend/fix_enum_migration.sql`

```sql
ALTER TABLE generation_task_variables 
MODIFY COLUMN source 
ENUM('USER_INPUT', 'SQL', 'API', 'AI_GENERATION', 'SYSTEM', 'IMAGE', 'VISION_AI') 
NOT NULL;
```

**执行方式**:
```bash

cd backend && python direct_alter.py

# 方式2: 直接使用mysql命令行
mysql -h 10.10.20.10 -P 24406 -u root -p123456 md_agent < backend/fix_enum_migration.sql
```

### 验证结果

## 2. 后台任务Session管理修复 ✅

### 问题描述
FastAPI的 `BackgroundTasks` 使用了request-scoped的数据库session，导致请求结束后session关闭，后台任务继续使用时出现：
```
PendingRollbackError: This Session's transaction has been rolled back
```

### 修复方案

**文件**: `backend/app/api/reports.py`

**修改1 - 函数签名** (第221-229行):
```python
# 修改前:
async def execute_report_generation(
    ...
    db_session: Session,  # ❌ 传入request-scoped session
    ...
)

# 修改后:
async def execute_report_generation(
    ...
    # ✅ 移除db_session参数
    ...
):
    # 在函数内创建新session
    from app.database import SessionLocal
    db_session = SessionLocal()
    
    try:
        # 执行任务逻辑
        ...
    finally:
        # 确保关闭session
        db_session.close()
```

**修改2 - 异常处理** (第419-455行):
```python
except Exception as e:
    try:
        # 回滚失败的事务
        db_session.rollback()
        
        # 更新任务状态
        task = db_session.query(GenerationTask).filter(...).first()
        if task:
            task.status = ReportStatus.FAILED
            db_session.commit()
    except Exception as commit_error:
        logger.error(f"Failed to update task status: {commit_error}")

finally:
    # 始终关闭session
    db_session.close()
    print(f"Database session closed for task {task_id}")
```

**修改3 - 调用点** (第485-494行):
```python
# 修改前:
background_tasks.add_task(
    execute_report_generation,
    ...
    db_session=db,  # ❌ 传递request-scoped session
    ...
)

# 修改后:
background_tasks.add_task(
    execute_report_generation,
    ...
    # ✅ 不再传递session
)
```

### 验证结果
日志中可见：
```

```

## 3. 数据库连接池优化 ✅

### 问题描述
缺少连接池配置，可能导致数据库连接过多或连接泄漏。

### 修复方案

**文件**: `backend/app/database.py`

**修改** (第16-23行):
```python
# 修改前:
engine_kwargs = {
    "pool_pre_ping": True,
    "pool_recycle": 3600,
    "echo": False,
}

# 修改后:
engine_kwargs = {
    "pool_pre_ping": True,      # 连接健康检查
    "pool_recycle": 3600,        # 1小时回收连接
    "pool_size": 5,              # 连接池大小（默认5）
    "max_overflow": 10,          # 最大溢出连接数
    "pool_timeout": 30,          # 获取连接超时（秒）
    "echo": False,               # SQL日志
}
```

### 配置说明
- **pool_size**: 核心连接池大小为5，适合中小型应用
- **max_overflow**: 最多允许额外10个连接，总计最多15个并发连接
- **pool_timeout**: 获取连接超时30秒，避免无限等待
- **pool_recycle**: 1小时回收连接，避免MySQL的 `wait_timeout` 问题

✅ **状态**: 已完成

## 4. Vision AI URL提取逻辑修复 ✅

### 问题描述
当 `image_config.output_format` 设置为 `url` 时，`VisionAiExecutor` 错误地将URL字符串构建成了base64 data URI，导致视觉AI模型拒绝该URL：
```
Error code: 400 - {'code': 20040, 'message': 'Image url should be a valid url or should like data:image/TYPE;base64,YOUR-BASE64-CONTENT'}
```

**问题根源**：
```python
# ImageExecutor 返回 (output_format: url)
{
    "data": "https://picsum.photos/400/300",  # URL字符串
    "mime_type": "image/png"
}

# VisionAiExecutor._extract_image_urls 错误处理
if "data" in image_data and image_data.get("mime_type"):
    # ❌ 无论data是什么，都构建成data URI
    urls.append(f"data:{mime_type};base64,{data}")
    # 结果: "data:image/png;base64,https://picsum.photos/400/300"
```

### 修复方案

**文件**: `backend/app/executors/vision_ai.py`

**修改** (第125-186行) - `_extract_image_urls` 方法:

```python
def _extract_image_urls(self, image_data: Any) -> List[str]:
    """智能提取图片URL，支持多种格式"""
    urls = []
    
    if isinstance(image_data, str):
        # 字符串：直接URL或Data URI
        if image_data.startswith(("http://", "https://", "data:")):
            urls.append(image_data)
    
    elif isinstance(image_data, dict):
        # 字典：ImageExecutor返回的格式
        if "data" in image_data:
            data = image_data["data"]
            
            # ✅ 检查data字段的内容类型
            if isinstance(data, str):
                # 如果是URL字符串或已经是data URI，直接使用
                if data.startswith(("http://", "https://", "data:")):
                    urls.append(data)
                # 如果是base64字符串且有mime_type，构建data URI
                elif image_data.get("mime_type"):
                    mime_type = image_data["mime_type"]
                    urls.append(f"data:{mime_type};base64,{data}")
            
            # 如果是bytes，编码为base64
            elif isinstance(data, bytes) and image_data.get("mime_type"):
                import base64
                mime_type = image_data["mime_type"]
                b64_data = base64.b64encode(data).decode('utf-8')
                urls.append(f"data:{mime_type};base64,{b64_data}")
        
        # 兜底：尝试从其他字段提取
        elif "url" in image_data:
            urls.append(image_data["url"])
        elif "markdown" in image_data:
            import re
            match = re.search(r'!\[.*?\]\((.*?)\)', image_data["markdown"])
            if match:
                urls.append(match.group(1))
    
    elif isinstance(image_data, list):
        # 列表：递归处理每个元素
        for item in image_data:
            urls.extend(self._extract_image_urls(item))
    
    return urls
```

### 修复效果

#### output_format: url
```python
# ImageExecutor 返回
{"data": "https://picsum.photos/400/300", "mime_type": "image/png"}

# ✅ 正确提取
urls = ["https://picsum.photos/400/300"]
```

#### output_format: base64
```python
# ImageExecutor 返回
{"data": "iVBORw0KGgoAAAANS...", "mime_type": "image/png"}

# ✅ 正确提取
urls = ["data:image/png;base64,iVBORw0KGgoAAAANS..."]
```

## 风险提示

1. **数据库迁移**:
   - ALTER TABLE期间可能短暂锁表
   - 建议在低峰期执行
   - 已有数据不受影响

2. **Session管理**:
   - 每个后台任务创建独立session
   - 异常情况下确保session关闭
   - 添加了rollback机制

3. **连接池限制**:
   - 高并发时可能达到连接池上限
   - 可根据实际负载调整 `pool_size` 和 `max_overflow`
