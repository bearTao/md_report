# 常量自动注入功能说明

## 功能概述

从本版本开始，所有 `source: constant` 类型的变量将自动预先执行并注入到执行上下文中，**无需在其他变量中显式声明依赖关系**。

## 实施日期
2025-10-29

## 核心改进

### 修改前（需要显式声明依赖）

```yaml
# 定义常量
api_base_url:
  type: string
  source: constant
  value: "http://10.10.20.10:5000"
  description: "API基础地址"

# 使用常量的变量 - 必须声明依赖
tech_users:
  type: object
  source: api
  dependencies: [api_base_url]  # ❌ 必须显式声明，容易遗漏
  api_config:
    endpoint: "{{api_base_url}}/api/users/query"
    method: POST
    body:
      department: "技术部"
  description: "技术部员工查询"
```

### 修改后（自动可用）

```yaml
# 定义常量
api_base_url:
  type: string
  source: constant
  value: "http://10.10.20.10:5000"
  description: "API基础地址"

# 使用常量的变量 - 无需声明依赖
tech_users:
  type: object
  source: api
  # ✅ 无需声明 dependencies，api_base_url 自动可用！
  api_config:
    endpoint: "{{api_base_url}}/api/users/query"
    method: POST
    body:
      department: "技术部"
  description: "技术部员工查询"
```

## 执行流程

### 三阶段执行模型

```
阶段 1: 预执行所有 CONSTANT 变量
   ├─ 并行执行所有常量
   ├─ 自动注入到 context
   └─ 失败不影响其他变量

阶段 2: 构建非常量变量的 DAG
   ├─ 只包含非常量变量
   ├─ 分析依赖关系
   └─ 确定执行顺序

阶段 3: 执行非常量变量
   └─ 按依赖顺序批量执行
```

### 执行示例

```yaml
# 变量定义
api_base_url: constant (value: "http://...")
min_salary: constant (value: 15000)
department: user_input
user_query: api (使用 api_base_url 和 min_salary)

# 实际执行顺序
1. api_base_url → 立即执行（阶段1）
2. min_salary → 立即执行（阶段1）
3. department → 等待用户输入（阶段3）
4. user_query → 在 department 之后执行（阶段3）
```

## 使用场景

### 场景 1：API 配置常量

```yaml
# 常量定义
api_base_url:
  type: string
  source: constant
  value: "http://10.10.20.10:5000"

api_timeout:
  type: number
  source: constant
  value: 30

# 在多个 API 变量中使用（无需声明依赖）
users_api:
  type: object
  source: api
  api_config:
    endpoint: "{{api_base_url}}/api/users"
    timeout: "{{api_timeout}}"

products_api:
  type: object
  source: api
  api_config:
    endpoint: "{{api_base_url}}/api/products"
    timeout: "{{api_timeout}}"
```

### 场景 2：业务常量

```yaml
# 业务常量
min_salary_standard:
  type: number
  source: constant
  value: 15000

max_salary_standard:
  type: number
  source: constant
  value: 50000

vat_rate:
  type: number
  source: constant
  value: 0.13

# 在 API 请求中使用（自动保持数字类型）
salary_query:
  type: object
  source: api
  api_config:
    endpoint: "http://api.example.com/salary/query"
    method: POST
    body:
      min_salary: "{{min_salary_standard}}"  # → 15000 (number)
      max_salary: "{{max_salary_standard}}"  # → 50000 (number)
      vat_rate: "{{vat_rate}}"              # → 0.13 (float)
```

### 场景 3：混合常量和动态变量

```yaml
# 常量
api_base_url:
  type: string
  source: constant
  value: "http://10.10.20.10:5000"

# 用户输入
department_name:
  type: string
  source: user_input
  required: true

min_salary_value:
  type: number
  source: user_input
  required: true

# API 调用
dynamic_query:
  type: object
  source: api
  dependencies: [department_name, min_salary_value]  # 只需声明非常量依赖
  api_config:
    endpoint: "{{api_base_url}}/api/users/query"  # 常量自动可用
    method: POST
    body:
      department: "{{department_name}}"
      min_salary: "{{min_salary_value}}"
```

## 技术细节

### 实现位置

文件：`backend/app/services/scheduler.py`

方法：`ExecutionScheduler.execute_all()`

### 关键特性

1. **自动执行**
   - 所有常量在任何其他变量之前执行
   - 无需手动管理执行顺序

2. **并行处理**
   - 多个常量并行执行
   - 提高初始化速度

3. **容错处理**
   - 常量执行失败不会中断其他变量
   - 只记录警告日志
   - 继续执行非常量变量

4. **类型保持**
   - 配合智能类型保持功能
   - 数字、布尔等类型自动保持

## 向后兼容性

### ✅ 完全兼容

1. **显式声明依然有效**
   ```yaml
   api_result:
     dependencies: [api_base_url]  # 显式声明也可以
   ```

2. **现有配置无需修改**
   - 已有的 dependencies 声明不会造成问题
   - 常量已经在 context 中，依赖检查会通过

3. **行为一致**
   - 对用户透明
   - 只是简化了配置

## 最佳实践

### ✅ 推荐做法

1. **常量用于固定值**
   ```yaml
   company_name:
     source: constant
     value: "XX科技有限公司"
   ```

2. **配置参数使用常量**
   ```yaml
   api_config_host:
     source: constant
     value: "10.10.20.10"
   
   api_config_port:
     source: constant
     value: 5000
   ```

3. **只声明非常量依赖**
   ```yaml
   my_api:
     dependencies: [user_input_var, another_api_var]
     # 不需要声明常量依赖
   ```

### ❌ 不推荐做法

1. **不要将动态数据定义为常量**
   ```yaml
   # ❌ 错误：用户输入不应该是常量
   user_department:
     source: constant
     value: "技术部"
   
   # ✅ 正确：应该是 user_input
   user_department:
     source: user_input
     required: true
   ```

2. **不要将需要计算的值定义为常量**
   ```yaml
   # ❌ 错误：需要查询数据库
   latest_salary:
     source: constant
     value: ???
   
   # ✅ 正确：应该是 sql 或 api
   latest_salary:
     source: sql
     sql_config: ...
   ```

## 故障排查

### 问题 1：常量值在其他变量中不可用

**检查项：**
1. 常量是否正确定义了 `value` 字段
2. 常量是否执行成功（查看日志）
3. 变量名是否正确（区分大小写）

**示例：**
```yaml
# 检查常量定义
api_base_url:
  source: constant
  value: "http://..."  # ✅ 确保有 value 字段

# 检查引用
endpoint: "{{api_base_url}}"  # ✅ 变量名完全匹配
```

### 问题 2：常量执行失败

**常见原因：**
- `value` 字段为 `None` 或缺失

**解决方案：**
```yaml
# ❌ 错误
my_constant:
  source: constant
  # 缺少 value 字段

# ✅ 正确
my_constant:
  source: constant
  value: "some_value"
```

### 问题 3：常量在日志中显示为失败

**日志示例：**
```
WARNING - Constant variable 'api_base_url' failed: ...
```

**处理：**
- 常量失败不会中断执行
- 检查常量定义是否正确
- 其他变量会继续执行

## 迁移指南

### 从旧配置迁移

如果你有现有配置显式声明了常量依赖，可以选择：

1. **保持不变**（推荐，向后兼容）
   ```yaml
   my_var:
     dependencies: [constant_var, other_var]  # 保持原样
   ```

2. **移除常量依赖**（可选，更简洁）
   ```yaml
   my_var:
     dependencies: [other_var]  # 移除常量依赖
   ```

两种方式都能正常工作，选择你喜欢的即可。
