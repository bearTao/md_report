## 🎯 实现功能

### 1. ✅ 模板嵌套/包含功能

**核心特性**:
- 支持 `{% include "template_id" %}` 语法嵌套其他模板
- 每个子模板独立执行其变量定义
- 子模板只能访问其自己元数据中定义的变量
- 自动检测并防止循环嵌套
- 支持多级嵌套（A包含B，B包含C）
- 错误处理优雅降级（显示错误信息而不中断）

**实现文件**:
- `backend/app/services/renderer.py` - 添加 `_resolve_includes()` 方法
- `backend/app/api/reports.py` - 在渲染前调用 `_resolve_includes()`
- `frontend/src/pages/templates/TemplateEdit.tsx` - 添加使用提示

**技术亮点**:
- 使用visited集合追踪已处理的template_id，防止循环嵌套
- 为每个子模板创建独立的ExecutionContext和Scheduler
- 正则表达式匹配：`{%\s*include\s+"([^"]+)"\s*%}`
- 反向处理matches避免索引位移问题

### 2. ✅ 调试界面功能

**核心特性**:
- 实时模板调试，无需保存
- 真实变量执行（SQL/API/AI）
- 左右分栏布局：编辑区 + 结果展示区
- Monaco编辑器支持Jinja2和YAML语法高亮
- 动态生成用户输入表单
- 变量执行详情展示（状态、耗时、错误）
- 支持从模板编辑页快速跳转

**实现文件**:

**后端**:
- `backend/app/api/debug.py` - 调试API端点
- `backend/app/schemas/api_schemas.py` - 调试相关Schema
- `backend/app/main.py` - 注册调试路由

**前端**:
- `frontend/src/pages/debug/TemplatePlayground.tsx` - 调试页面组件
- `frontend/src/api/debug.ts` - 调试API调用
- `frontend/src/App.tsx` - 添加调试路由
- `frontend/src/pages/templates/TemplateEdit.tsx` - 添加"调试模板"按钮

**技术亮点**:
- 使用React Router的state传递模板内容和元数据
- 实时YAML解析和错误提示
- 动态表单生成（根据metadata自动创建字段）
- Markdown实时预览（ReactMarkdown）
- 变量执行结果表格化展示

### 新增API

#### POST /api/debug/render
调试模板渲染

**请求体**:
```json
{
  "template_content": "# {{title}}",
  "metadata_yaml": "title:\n  type: string\n  source: user_input\n  ...",
  "user_inputs": {
    "title": "测试"
  }
}
```

**响应**:
```json
{
  "success": true,
  "rendered_markdown": "# 测试",
  "variables": [
    {
      "variable_name": "title",
      "status": "success",
      "value": "测试",
      "duration_ms": 5,
      "error_message": null
    }
  ],
  "error": null
}
```

## 🎯 使用示例

### 示例1：模板嵌套

4. 查看右侧的渲染结果和变量执行详情

## ⚙️ 技术架构

### 模板嵌套流程

```
1. 主模板内容
   ↓
2. _resolve_includes() 解析{% include %}标签
   ↓
3. 递归处理：
   a. 从数据库加载子模板
   b. 检查循环嵌套
   c. 递归解析子模板中的include
   d. 创建独立的ExecutionContext
   e. 执行子模板变量
   f. 渲染子模板
   g. 替换include标签
   ↓
4. 返回完全解析的模板内容
   ↓
5. 主模板渲染
```

### 调试台流程

```
前端:
1. 用户编辑模板内容和元数据
   ↓
2. 解析YAML → 生成用户输入表单
   ↓
3. 用户填写输入 → 点击"执行调试"
   ↓
4. POST /api/debug/render

后端:
5. 解析metadata YAML
   ↓
6. 创建临时ExecutionContext
   ↓
7. 执行所有变量（真实SQL/API/AI调用）
   ↓
8. 解析includes（如有）
   ↓
9. 渲染模板
   ↓
10. 返回结果 + 变量执行详情

前端:
11. 显示渲染结果
    ↓
12. 显示变量执行表格
```

## 🔐 安全考虑

### 模板嵌套
- ✅ 循环嵌套检测防止无限递归
- ✅ 子模板变量作用域隔离
- ✅ 错误信息不暴露敏感数据

### 调试台
- ⚠️ **真实执行警告**：调试会消耗API配额和数据库资源
- 🔒 未实现调用频率限制（计划在P1.2中实现）
- 🔒 调试配置不会保存到数据库
- 🔒 建议在生产环境限制调试API访问权限

## 📈 性能影响

### 模板嵌套
- **变量执行**：每个子模板独立执行，SQL/API/AI调用次数增加
- **建议**：
  - 避免在子模板中执行耗时查询
  - 合理使用缓存（P1.2功能）
  - 控制嵌套层级（建议≤3层）

### 调试台
- **真实调用**：所有变量都会真实执行
- **建议**：
  - 在测试环境使用
  - AI调试谨慎使用（消耗API配额）
  - 实施调用频率限制（P1.2）

---

## 🐛 已知限制

### 模板嵌套
1. ❌ 子模板无法访问父模板变量（设计如此，保证作用域隔离）
2. ❌ 同一模板被多次包含时，每次都会重新执行变量（计划优化）

### 调试台
1. ❌ 没有调用频率限制（P1.2）
2. ❌ 不支持调试配置保存（设计如此）
3. ❌ 大型报告渲染可能较慢
