# 同名变量冲突问题修复总结

## 问题描述

用户发现了一个严重问题：当嵌套模板中的不同模板有同名变量时（例如模板A和模板B都有变量 `title1`），会导致其中一个模板的变量一直处于"执行中"状态，无法完成。

### 问题示例

```
模板A (template_a):
  - 变量: title1
  - 变量: content1

模板B (template_b):
  - 变量: title1  ⚠️ 与模板A的变量同名！
  - 变量: content2
```

当同时执行这两个模板时：
- 模板A的 `title1` 开始执行，创建数据库记录
- 模板B的 `title1` 开始执行，但更新时会找到模板A的记录（因为只按 task_id 和 variable_name 查找）
- 结果：模板A的 `title1` 记录被覆盖，显示为模板B的状态
- 模板B的 `title1` 没有自己的记录，一直处于"执行中"

## 问题根本原因

### 1. 数据库查询条件不完整

在更新变量执行状态时，查询条件只使用了 `(task_id, variable_name)`：

```python
# ❌ 错误的查询 - 缺少 template_id
var_record = db_session.query(GenerationTaskVariable).filter(
    GenerationTaskVariable.task_id == task_id,
    GenerationTaskVariable.variable_name == var_name
).first()
```

**问题**：如果有多个模板有同名变量，会找到第一个匹配的记录，导致错误地更新了其他模板的变量。

if len(vars) > 1:

## 技术要点

### 1. 变量唯一性
在同一个任务中，变量的唯一标识应该是：`(task_id, template_id, variable_name)`

而不是：`(task_id, variable_name)`

### 2. 数据库设计建议
虽然当前 `GenerationTaskVariable` 表使用自增ID作为主键，但建议添加唯一约束：

```sql
ALTER TABLE generation_task_variables 
ADD UNIQUE INDEX idx_task_template_variable (task_id, template_id, variable_name);
```

这样可以从数据库层面防止重复记录。

### 3. 重试功能的限制
当有多个同名变量时，重试功能会返回错误，要求前端明确指定要重试哪个变量。

未来改进：前端可以在重试请求中包含 `template_id` 参数。

### 4. ExecutionContext 的作用域
每个模板（主模板和子模板）都有自己的 `ExecutionContext`，变量是相互独立的。

但在数据库查询时，如果不包含 `template_id`，就会导致混淆。

## 潜在问题与未来改进

### 1. 变量依赖跨模板的情况
当前修复假设每个模板的变量是独立的。如果存在跨模板的变量依赖，可能需要：
- 在模板元数据中明确指定依赖的模板和变量
- 使用完全限定名称，如 `template_a.title1`

### 2. 重试功能的用户体验
当有同名变量时，重试会失败并提示用户。未来可以：
- 在前端显示完整的 `template_path`
- 允许用户选择要重试哪个模板的变量
- API 接受 `template_id` 参数

### 3. WebSocket 广播
WebSocket 事件中只包含 `variable_name`，如果前端有多个同名变量，可能需要：
- 在事件中包含 `template_id` 或 `template_path`
- 前端根据模板信息区分不同的变量
