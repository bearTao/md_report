|--------|------|------|------|---------|
| type | string | 是 | 变量数据类型 | "string", "number", "boolean", "array", "object" |

### SQL数据库变量 (source: "sql")

| 字段名 | 类型 | 必填 | 描述 | 示例值 |
|--------|------|------|------|---------|
| sql_config | object | 是 | SQL查询配置 |  |
| sql_config.connection | string | 是 | 数据库连接名称 | "user_db", "project_db" |
| sql_config.query | string | 是 | SQL查询语句 | "SELECT name FROM users WHERE id = {{user_id}}" |
| sql_config.parameters | array | 条件必填 | 参数化查询的变量名（SQL中使用`:param`时必填） | ["user_id"], ["project_id", "start_date"], [] |
| sql_config.timeout | number | 否 | 查询超时时间(秒) | 10, 30 |
| sql_config.result_mode | string | 否 | 结果返回模式 | "first_row", "all_rows", "first_value", "first_column", "auto" (默认) |

**parameters 字段说明：**

`parameters` 字段用于声明哪些变量需要通过参数化查询（`:param_name`）传递给数据库：

- **必填情况**：SQL 中使用了 `:param_name` 占位符时，必须在 `parameters` 中声明
- **可选情况**：SQL 中只使用 `{{variable}}` 插值，或完全静态查询时，可以为空数组 `[]` 或省略
- **作用**：防止 SQL 注入，数据库驱动会自动处理类型转换和引号

**与 dependencies 的区别：**
- `dependencies`（变量级别）：声明执行顺序依赖，包括所有使用到的变量（插值 + 参数化）
- `parameters`（SQL 级别）：声明参数化查询的参数，只包括 `:param_name` 的变量
- **关系**：`parameters` 必须是 `dependencies` 的子集

**示例对比：**
```yaml
# 场景1：只用参数化（最常见）
query: "SELECT * FROM users WHERE id = :user_id"
dependencies: ["user_id"]  # 依赖 user_id
parameters: ["user_id"]    # 参数化 user_id
# 两者相同

# 场景2：混合使用
query: "SELECT {{columns}} FROM {{table}} WHERE id = :user_id"
dependencies: ["columns", "table", "user_id"]  # 依赖3个变量
parameters: ["user_id"]                        # 只参数化1个
# parameters 是 dependencies 的子集

# 场景3：只用插值（不推荐，有注入风险）
query: "SELECT * FROM {{table_name}}"
dependencies: ["table_name"]  # 依赖 table_name
parameters: []                # 无参数化
```

**result_mode 说明：**
- `first_row`: 返回第一行作为对象 `{col1: val1, col2: val2}`
- `all_rows`: 返回所有行作为数组 `[{row1}, {row2}, ...]`
- `first_value`: 返回第一行第一列的值（标量）
- `first_column`: 返回第一列的所有值 `[val1, val2, ...]`
- `auto`: 根据 `type` 自动判断（默认行为）
  - `type="object"`: 单行返回对象，多行返回数组
  - `type="array"`: 返回所有行
  - `type="string/number/boolean"`: 返回第一行第一列

### API接口变量 (source: "api")

| 字段名 | 类型 | 必填 | 描述 | 示例值 |
|--------|------|------|------|---------|
| api_config | object | 是 | API调用配置 |  |
| api_config.endpoint | string | 是 | API接口地址 | "https://api.weather.com/v1/current" |
| api_config.method | string | 否 | HTTP方法 | "GET" (默认), "POST", "PUT", "DELETE" |
| api_config.headers | object | 否 | 请求头配置 | {"Authorization": "Bearer {{api_key}}"} |
| api_config.parameters | object | 否 | URL参数（支持任意类型） | {"location": "{{city}}", "limit": 10} |
| api_config.body | object | 否 | 请求体数据(POST/PUT) | {"symbols": "{{stock_symbols}}"} |
| api_config.response_mapping | string/object | 否 | 响应数据映射规则 | 见下方response_mapping说明 |
| api_config.timeout | number | 否 | 请求超时时间(秒) | 10 (默认) |
| api_config.cache_ttl | number | 否 | 缓存时间(秒) | 1800 (30分钟) |
| api_config.retry_count | number | 否 | 重试次数 | 0 (默认不重试), 3 |
| api_config.retry_status_codes | array | 否 | 需要重试的HTTP状态码 | [429, 500, 502, 503, 504] (默认) |
| api_config.retry_backoff | number | 否 | 重试间隔基数(秒) | 1.0 (默认) |

**类型保持说明**：

`parameters` 和 `body` 中的变量插值会智能保持原始类型：

**纯变量引用** - 保持原始类型：
```yaml
api_config:
  body:
    count: "{{total_count}}"      # total_count=15 → 发送 15 (number)
    active: "{{is_active}}"       # is_active=true → 发送 true (boolean)
```

**模板字符串** - 返回字符串：
```yaml
api_config:
  body:
    message: "共{{count}}条"      # → "共15条" (string)
```

**判断规则**：
- 整个字符串只有 `{{var}}` → 保持原始类型
- 包含其他文本或多个变量 → 转换为字符串

#### response_mapping 使用说明

`response_mapping` 支持三种模式，根据值的类型自动判断：

**模式1：完整响应（不映射）**
```yaml
# 方式1：不写 response_mapping 字段
api_config:
  endpoint: "http://api.example.com/data"
  
# 方式2：设为 null
api_config:
  endpoint: "http://api.example.com/data"
  response_mapping: null
```
返回 API 的完整原始响应。

**模式2：单路径提取（字符串）**
```yaml
# 提取嵌套对象
api_config:
  endpoint: "http://api.example.com/weather"
  response_mapping: "data.current"  # 字符串

# 提取数组
api_config:
  endpoint: "http://api.example.com/products"
  response_mapping: "data.items"  # 返回纯数组

# 提取基础类型
api_config:
  endpoint: "http://api.example.com/version"
  response_mapping: "version"  # 返回字符串 "2.5.1"
```
使用 JMESPath 语法提取单个路径，返回值可以是任意类型（对象/数组/字符串/数字）。

**模式3：多字段映射（对象）**
```yaml
api_config:
  endpoint: "http://api.example.com/weather"
  response_mapping:  # 对象：映射多个字段
    temperature: "data.current.temp"
    condition: "data.current.condition"
    latitude: "coord.lat"
```
从不同路径提取多个字段并重组为新对象。

#### JMESPath 语法支持

系统使用 JMESPath 进行强大的路径提取，支持：

- **简单路径**: `"data.items"`
- **数组索引**: `"items[0]"`, `"data.products[0].name"`
- **数组切片**: `"items[:3]"`, `"items[1:5]"`
- **投影（提取所有）**: `"items[*].name"` - 提取所有name字段
- **条件过滤**: `"items[?price > \`100\`]"` - 筛选价格>100的项
- **函数**: `"length(items)"`, `"sum(items[*].price)"`

详见 [JMESPath 文档](https://jmespath.org/)

#### 重试机制说明

当配置 `retry_count > 0` 时，系统会在以下情况自动重试：
- HTTP 状态码在 `retry_status_codes` 列表中
- 请求超时
- 重试间隔采用线性退避：第n次重试等待 `retry_backoff * n` 秒

示例：
```yaml
api_config:
  endpoint: "http://api.unstable.com/data"
  retry_count: 3  # 最多重试3次
  retry_status_codes: [429, 500, 503]  # 这些状态码会重试
  retry_backoff: 2.0  # 第1次等2秒，第2次等4秒，第3次等6秒
```


### 常量变量 (source: "constant")

| 字段名 | 类型 | 必填 | 描述 | 示例值 |
|--------|------|------|------|---------|
| value | any | 是 | 常量的固定值 | 15000, "北京", true, [1,2,3] |

**使用场景**：
- 业务常量：薪资标准、税率、折扣率等
- 配置参数：API地址、公司信息、系统参数等
- 固定参数：在多个变量中共享的不变值

**特点**：
- 值在模板定义时固定，不会动态生成
- **自动预执行**：常量总是在所有其他变量之前自动执行
- **无需声明依赖**：其他变量使用常量时无需在 `dependencies` 中声明
- 支持所有数据类型（number, string, boolean, array, object）
- 执行速度最快（直接返回值）
- 配合智能类型保持，数字/布尔等类型自动保持

**示例**：

```yaml
# 业务常量
min_salary_standard:
  type: number
  source: constant
  description: "公司最低薪资标准"
  value: 15000

# 配置常量
api_base_url:
  type: string
  source: constant
  description: "API基础地址"
  value: "http://10.10.20.10:5000"

# 税率常量
vat_rate:
  type: number
  source: constant
  description: "增值税率"
  value: 0.13

# 布尔常量
enable_cache:
  type: boolean
  source: constant
  description: "是否启用缓存"
  value: true

# 数组常量
major_cities:
  type: array
  source: constant
  description: "主要城市列表"
  value: ["北京", "上海", "深圳", "杭州"]

# 对象常量
default_config:
  type: object
  source: constant
  description: "默认配置"
  value:
    timeout: 30
    retry: 3
    cache_ttl: 1800
```

**在其他变量中使用常量**：

```yaml
# 定义常量
api_base_url:
  type: string
  source: constant
  value: "http://10.10.20.10:5000"

min_salary:
  type: number
  source: constant
  value: 15000

# 在 API 调用中引用常量
user_query:
  type: object
  source: api
  # ✅ 无需声明常量依赖！常量自动可用
  # dependencies: []  
  api_config:
    endpoint: "{{api_base_url}}/api/users"  # 常量自动可用
    method: POST
    body:
      min_salary: "{{min_salary}}"  # 引用常量，会保持数字类型
      department: "技术部"
```

**自动注入说明**：

常量变量会自动预先执行并注入到执行上下文：

- ✅ **无需声明依赖**：使用常量时不需要在 `dependencies` 中声明
- ✅ **自动优先执行**：常量总是在其他变量之前执行
- ✅ **类型保持**：数字/布尔等类型自动保持

**示例**：

```yaml
# 定义常量
api_base_url:
  type: string
  source: constant
  value: "http://10.10.20.10:5000"

# 使用常量
tech_users:
  type: object
  source: api
  # ✅ 无需声明 dependencies: [api_base_url]
  api_config:
    endpoint: "{{api_base_url}}/api/users"
```

## 配置对象详细说明

### Schema 数据结构定义

**使用说明：**

`schema` 字段仅用于 **AI 生成变量**和 **Vision AI 变量**，作用是：

1. **指导 AI 生成**：告诉 AI 应该返回什么结构的数据
2. **验证 AI 输出**：检查 AI 返回的数据是否符合预期结构

**对于其他变量类型：**
- **SQL 变量**：`schema` 字段不会被使用，可以省略（仅作文档用途）
- **API 变量**：`schema` 字段不会被使用，可以省略（仅作文档用途）
- **用户输入变量**：`schema` 字段不会被使用，可以省略

**推荐做法：**
- ✅ AI/Vision AI 变量：**必须配置** `schema`，确保输出结构正确
- ⚠️ SQL/API 变量：**可以省略** `schema`，减少配置冗余

| 字段名 | 类型 | 必填 | 描述 | 示例值 |
|--------|------|------|------|---------|
| type | string | 是 | 数据类型 | "object", "array", "string", "number" |
| properties | object | 对象必填 | 对象属性定义 | {"name": {"type": "string"}} |
| items | object | 数组必填 | 数组元素定义 | {"type": "object", "properties": {...}} |
| format | string | 否 | 数据格式 | "date", "email", "url" |
| enum | array | 否 | 枚举值列表 | ["高", "中", "低"] |

| item_mapping | object | 数组映射 | 数组元素字段映射 | {"symbol": "symbol", "price": "price"} |

### System Config 系统配置

## 数据类型说明

### type 字段可选值

| 类型值 | 描述 | 使用场景 |
|--------|------|----------|

| 来源值 | 描述 | 使用场景 |
|--------|------|----------|
| "user_input" | 用户输入 | 需要用户填写或选择的数据 |

| "api" | 接口调用 | 从外部API获取的数据 |

| "image" | 图片获取 | 从HTTP API获取图片数据 |
| "vision_ai" | 视觉AI分析 | 使用视觉大模型分析图片内容 |
| "system" | 系统生成 | 系统自动生成的数据（时间、ID等） |

report_title:

  required: true
  description: "报告标题"

### SQL数据库变量示例

```yaml
# 示例1：获取单个用户档案（返回对象）
user_profile:
  type: "object"
  source: "sql"
  required: false
  default: {}
  description: "用户档案信息"
  dependencies: ["user_id"]  # 声明依赖：用于执行顺序
  sql_config:
    connection: "user_db"
    query: "SELECT name, email, department, role FROM users WHERE id = :user_id"  # 使用 :user_id 参数化
    parameters: ["user_id"]  # 必须声明：因为 SQL 中使用了 :user_id
    timeout: 10
    result_mode: "first_row"  # 明确返回第一行作为对象


# 示例2：获取用户列表（返回数组）
team_members:
  type: "array"
  source: "sql"
  required: false
  default: []
  description: "团队成员列表"
  dependencies: ["department_id"]
  sql_config:
    connection: "user_db"
    query: "SELECT name, email, role FROM users WHERE department_id = :department_id"
    parameters: ["department_id"]
    timeout: 10
    result_mode: "all_rows"  # 返回所有行

# 示例3：获取总计数（返回标量）
total_sales:
  type: "number"
  source: "sql"
  required: false
  default: 0
  description: "总销售额"
  dependencies: ["start_date", "end_date"]
  sql_config:
    connection: "analytics_db"
    query: "SELECT SUM(amount) FROM sales WHERE date BETWEEN :start_date AND :end_date"
    parameters: ["start_date", "end_date"]
    timeout: 10
    result_mode: "first_value"  # 返回第一行第一列的值

# 示例4：获取用户ID列表（返回单列数组）
# 注意：这个例子是静态查询，无需 parameters
user_ids:
  type: "array"
```

## 常见问题和解决方案

### ⚠️ 核心警告：字典方法名冲突

这是最常见、最容易忽视的问题！

**问题本质**：
Python 字典有 40+ 个内置方法，在 Jinja2 模板中使用点号访问会**优先访问方法**而不是键值！

**最危险的字段名**：
- 🔴 `items`, `keys`, `values` - 极高风险，最常用
- 🔴 `get`, `pop`, `update`, `clear`, `copy` - 高风险

**快速解决**：
```jinja2
{# ❌ 错误：会访问方法 #}
{{ order.items | length }}

{# ✅ 正确：使用方括号 #}
{{ order["items"] | length }}
```

**推荐做法**：
1. **API 返回数据**：直接用方括号 `dict["items"]`（简单直接）
2. **SQL 查询**：使用别名 `"items" AS order_items`（提高可读性）
3. **设计阶段**：避免使用这些字段名

详见《报告模板编写规范指南》第4节"陷阱8"。

---

### 问题1：字典方法名冲突导致渲染失败

**错误信息**：
```
TypeError: object of type 'builtin_function_or_method' has no len()
```

**原因**：
SQL 返回的字典包含与 Python 字典方法重名的键（如 `items`, `keys`, `values`）

**解决方案**：
1. **推荐**：在模板中使用方括号
```jinja2
{{ order["items"] | length }}
```

2. **SQL场景**：在 SQL 中使用别名
```sql
SELECT "items" AS order_items FROM orders
```

详见《报告模板编写规范指南》第4节"陷阱8"。

### 问题2：API body 中数字被转换为字符串

**现象**：
```yaml
body:
  min_salary: "{{min_salary_value}}"  # 发送 "15000" 而不是 15000
```

**后端报错**：
```json
{"error": "Invalid type for min_salary, expected number but got string"}
```

**解决方案**：
从 2025-10-29 版本开始，纯变量引用会自动保持原始类型，无需额外配置。

如果使用旧版本，可以：
```yaml
# 方案1：使用 SQL 转换
sql_config:
  query: "SELECT CAST(min_salary AS INTEGER) AS min_salary_int"

# 方案2：使用 AI 生成正确格式的 JSON
```

### 问题4：为什么方括号能解决冲突？

**技术原理**：

```python
# Python 中的两种访问方式
d = {'items': [1, 2, 3]}

d.items        # 访问字典的 items() 方法
d['items']     # 访问字典的 'items' 键
```

在 Jinja2 中：
- `{{ dict.field }}` → 按顺序查找：对象属性 → 字典方法 → 字典键
- `{{ dict["field"] }}` → 直接访问字典键（跳过前面的步骤）

**因此方括号能直接访问键值，不会被方法拦截！**

---

### 问题5：什么时候用别名，什么时候用方括号？

**决策树**：

```
数据来源是什么？
  ├─ SQL 查询
  │   └─ 推荐：使用别名（提高可读性）
  │        SELECT "items" AS order_items
  │
  ├─ API 调用（无法控制字段名）
  │   └─ 推荐：使用方括号（简单直接）
  │        {{ result["items"] }}
  │
  └─ AI 生成
      └─ 推荐：使用方括号
           {{ data["items"] }}
```

**总结**：能控制字段名用别名，不能控制用方括号

### 问题6：如何快速检查字段名是否冲突？

**方法1：Python 检查**
```python
field_name = "items"
if hasattr(dict, field_name):
    print(f"⚠️ 字段名 '{field_name}' 与字典方法冲突！")
    print(f"建议改为: {field_name}_data 或 {field_name}_list")
```

**方法2：常用冲突列表**
```python
# 记住这些高危字段名
high_risk_fields = ['items', 'keys', 'values', 'get', 'pop', 
                    'update', 'clear', 'copy']
```

**方法3：快速测试**
```python
# 在 Python 中快速测试
>>> d = {'items': [1, 2, 3]}
>>> d.items
<built-in method items...>  # 是方法！危险！
>>> d['items']
[1, 2, 3]  # 是数据，正确
```
