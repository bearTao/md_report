## 🔧 技术实现

### 变量隔离机制

```
用户提交inputs:
{
    "main_template": {"title": "主标题"},
    "sub_template": {"title": "子标题"}
}
    ↓
后端处理:
1. 主模板执行时，context.user_inputs = {"title": "主标题"}
2. 渲染主模板: "# {{title}}" → "# 主标题"
3. 遇到 {% include "sub_template" %}
4. 子模板执行时，context.user_inputs = {"title": "子标题"}
5. 渲染子模板: "## {{title}}" → "## 子标题"
6. 替换到主模板中
    ↓
最终结果:
# 主标题
## 子标题
```

### 前端扫描逻辑

```typescript
// 1. 正则匹配 include 标签
const includePattern = /{%\s*include\s+"([^"]+)"\s*%}/g;
const templateIds = [...content.matchAll(pattern)].map(m => m[1]);

// 2. 加载所有被include的模板
for (const tid of templateIds) {
    const temp = await getTemplate(tid);
    includedTemplates.push(temp);
}

// 3. 为每个模板生成Card section
{includedTemplates.map(temp => (
    <Card title={`子模板: ${temp.name}`}>
        {temp.variables.map(...)}
    </Card>
))}
```

## ⚠️ 重要说明

### 变量隔离规则

1. **每个模板的变量完全独立**
   - 主模板的 `title` 和子模板的 `title` 是两个不同的变量
   - 即使变量名相同，也不会相互影响

2. **生成报告时分别填写**
   - 表单会为每个模板显示一个Card section
   - 每个section中填写该模板的变量

3. **向后兼容**
   - 没有include的模板仍使用原来的扁平结构
   - 已有的报告生成流程不受影响

### 调试功能注意事项

1. **真实执行**：调试会消耗真实资源
   - SQL: 连接数据库
   - API: 发送HTTP请求
   - AI: 调用LLM（**消耗API配额**）

2. **不保存**：调试配置不会保存到数据库
