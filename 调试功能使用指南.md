# 调试功能使用指南

## 📖 功能概述

模板调试台（Template Playground）是一个强大的实时调试工具，允许您在不保存模板的情况下，直接测试模板渲染效果。所有变量将使用**真实的SQL/API/AI调用**执行，帮助您快速验证模板配置。

## 🚀 访问调试台

### 方法1：从模板编辑页快速进入

1. 打开任意模板的编辑页面
2. 点击"**调试模板**"按钮（🐛图标）
3. 系统会自动将当前的模板内容和元数据带入调试台

### 方法2：直接访问

在浏览器中访问：`http://localhost:5173/debug`

## 🎨 界面布局

调试台采用左右分栏布局：

### 左侧：编辑区
1. **模板内容编辑器**
   - 使用Monaco编辑器，支持Jinja2语法高亮
   - 支持代码补全和语法检查

2. **变量元数据编辑器**
   - YAML格式
   - 实时语法检查
   - 错误提示

3. **用户输入表单**
   - 自动根据元数据生成
   - 只显示 `source: user_input` 类型的变量
   - 支持必填项验证

4. **执行按钮**
   - 点击后开始真实执行所有变量
   - 显示执行进度

### 右侧：结果展示区
1. **渲染结果面板**
   - 实时预览Markdown渲染效果
   - 支持滚动查看长报告

2. **变量执行详情表格**
   - 显示每个变量的执行状态
   - 执行时间统计
   - 错误信息（如有）

## 📝 使用步骤

### 步骤1：编写或粘贴模板内容

在"模板内容"编辑器中输入Jinja2模板，例如：

```jinja2
# {{title}}

## 数据分析

### SQL查询结果
总记录数：{{record_count}}

### API数据
天气信息：{{weather_info}}

### AI生成
{{ai_summary}}
```

### 步骤2：配置变量元数据

在"变量元数据"编辑器中定义所有变量，例如：

```yaml
title:
  type: string
  source: user_input
  required: true
  description: 报告标题
  ui_config:
    input_type: text
    placeholder: 请输入报告标题

record_count:
  type: number
  source: sql
  description: 记录总数
  sql_config:
    connection_name: default
    query: "SELECT COUNT(*) as count FROM users"

weather_info:
  type: string
  source: api
  description: 天气信息
  api_config:
    method: GET
    endpoint: "https://api.weatherapi.com/v1/current.json"
    parameters:
      key: "your_api_key"
      q: "Beijing"
    response_mapping:
      path: "current.condition.text"

ai_summary:
  type: string
  source: ai_generation
  description: AI生成的总结
  dependencies:
    - record_count
  ai_config:
    model: "gpt-3.5-turbo"
    prompt_template: |
      基于以下数据生成一个简短的总结：
      用户总数：{{record_count}}
    temperature: 0.7
  schema:
    type: string
```

### 步骤3：填写用户输入

系统会自动根据元数据生成用户输入表单。填写所有必填字段，例如：
- **报告标题**: `2024年1月数据报告`

### 步骤4：执行调试

点击"**执行调试**"按钮，系统将：

1. ✅ 验证元数据YAML格式
2. ✅ 解析变量依赖关系
3. ✅ 按依赖顺序执行所有变量：
   - 执行SQL查询（真实数据库连接）
   - 调用API接口（真实HTTP请求）
   - 调用AI模型（真实LLM调用，**会消耗API配额**）
4. ✅ 渲染模板
5. ✅ 显示结果和详情

### 步骤5：查看结果

- **渲染结果面板**：查看最终生成的Markdown报告
- **变量执行详情**：
  - 绿色勾号✅ = 成功
  - 红色叉号❌ = 失败
  - 查看每个变量的执行时间
  - 查看错误信息（如有）

## 🎯 使用场景

### 场景1：快速验证模板语法
在保存模板前，先在调试台测试Jinja2语法是否正确。

### 场景2：测试变量配置
验证SQL查询、API调用、AI配置是否正确。

### 场景3：调试复杂依赖
当变量之间有复杂依赖关系时，通过调试台查看执行顺序和中间结果。

### 场景4：性能测试
查看每个变量的执行时间，找出性能瓶颈。

### 场景5：测试嵌套模板
测试包含 `{% include "template_id" %}` 的模板是否正确渲染。

## ⚙️ 高级功能

### 1. 实时YAML验证

元数据编辑器会实时检查YAML语法，如有错误会显示红色警告：

```
❌ YAML 解析错误
unexpected end of stream
```

### 2. 动态表单生成

用户输入表单会随元数据变化自动更新：
- 添加新的 `user_input` 变量 → 表单自动添加字段
- 删除变量 → 表单字段自动移除
- 修改 `required` → 必填规则自动更新

### 3. 错误详情展示

如果变量执行失败，详情表格会显示具体错误信息，例如：

| 变量名 | 状态 | 执行时间 | 结果 |
|--------|------|----------|------|
| record_count | ❌ 失败 | 1523ms | Table 'users' doesn't exist |

### 4. 模板嵌套测试

调试台完整支持 `{% include "template_id" %}` 语法，可以测试嵌套模板的渲染效果。

## ⚠️ 重要提醒

### ⚡ 真实执行
调试台执行的所有变量调用都是**真实的**：
- SQL查询会真实连接数据库
- API会发送真实HTTP请求
- AI会调用真实的LLM（**消耗API配额**）

**建议**：
- 在生产环境谨慎使用AI变量调试
- SQL查询使用只读数据库连接
- API调用注意频率限制

### 🔒 调试配置不会保存
调试台的所有配置和结果都是临时的，不会保存到数据库。如需保存，请返回模板编辑页面点击"保存模板"。

### 🚫 并发限制
为防止滥用，调试API可能设置调用频率限制（P1.2功能）。

## 🛠️ 故障排查

### 问题1：执行按钮无响应
**可能原因**：
- YAML格式错误
- 必填的用户输入未填写

**解决方案**：
- 检查元数据编辑器上方的错误提示
- 确保所有必填字段已填写

### 问题2：变量执行失败
**可能原因**：
- SQL连接未配置
- API密钥错误
- AI配置错误

**解决方案**：
- 查看变量执行详情表格中的错误信息
- 检查数据库连接配置（设置 → 数据库连接）
- 检查AI配置（设置 → AI配置）

### 问题3：渲染结果为空
**可能原因**：
- 模板内容为空
- 所有变量都执行失败

**解决方案**：
- 检查模板内容编辑器
- 查看变量执行详情，修复失败的变量

### 问题4：include标签未渲染
**可能原因**：
- 子模板ID不存在
- 子模板有循环嵌套

**解决方案**：
- 在模板列表页确认子模板ID
- 检查是否存在循环依赖

## 💡 调试技巧

### 技巧1：从简单开始
先测试最基本的用户输入变量，再逐步添加SQL、API、AI变量。

### 技巧2：使用默认值
在元数据中为变量设置 `default` 值，避免执行失败时整个报告无法生成。

```yaml
record_count:
  type: number
  source: sql
  default: 0  # 如果SQL执行失败，使用0
  sql_config:
    # ...
```

### 技巧3：分步测试
1. 先测试模板语法（只用user_input变量）
2. 再测试SQL变量
3. 最后测试API和AI变量

### 技巧4：查看执行时间
通过执行时间判断性能瓶颈，优化慢查询。
