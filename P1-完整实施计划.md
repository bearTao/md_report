## 数据库Schema变更

### 需要创建的表

```sql
CREATE TABLE execution_logs (
  id BIGSERIAL PRIMARY KEY,
  task_id VARCHAR(50) NOT NULL,
  variable_name VARCHAR(200),
  level VARCHAR(20) NOT NULL,
  message TEXT NOT NULL,
  context_json JSON,
  created_at TIMESTAMP NOT NULL DEFAULT NOW(),
  INDEX idx_task_id (task_id),
  INDEX idx_variable (task_id, variable_name),
  INDEX idx_level (level),
  INDEX idx_created_at (created_at)
);
```

### 实施方式建议

**方式1：代码自动创建**（推荐开发环境）
```python
# 在 backend/app/main.py 启动时
from app.database import engine, Base
from app.models.db_models import ExecutionLog  # 触发导入

@app.on_event("startup")
async def startup_event():
    Base.metadata.create_all(bind=engine)
```

**方式2：Alembic迁移**（推荐生产环境）
```bash
cd backend
alembic revision --autogenerate -m "Add execution_logs table"
alembic upgrade head
```

**方式3：手动SQL**（您完全控制）
- 我提供SQL脚本
- 您手动执行

**推荐**：开发用方式1，生产用方式2

## 实施建议

### 选项A：分阶段实施（推荐）
- **优点**：每个阶段可独立测试和验证，降低风险
- **缺点**：总时间稍长

### 选项B：并行开发
- **优点**：时间最短
- **缺点**：测试复杂度高，可能相互影响

### 选项C：一次性全部实施
- **优点**：一气呵成
- **缺点**：调试困难，问题定位难

**我推荐选项A**：先实施P1.1，测试通过后再进行P1.2和P1.3。
